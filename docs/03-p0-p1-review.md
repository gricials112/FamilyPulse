# FamilyPulse 需求严审与优化记录

版本: 0.3  
日期: 2026-05-31

## P0 问题

### P0-1 家庭健康数据隐私边界不清

问题: 照护记录、家庭成员和复查安排属于家庭隐私数据。若首版没有家庭 membership 校验, 任意用户可能访问他人资料。  
优化:

- 所有 `/api/families/{familyId}/...` 接口必须读取请求用户并校验 membership。
- 服务端测试覆盖“非成员访问 403”。
- 服务端测试覆盖“未登录访问 401”。

### P0-2 一键动作可能误点/重复点导致家庭误判

问题: 老人误点或多次点击会直接影响子女判断。  
优化:

- App 完成后 10 秒内显示撤销。
- 服务端使用 `elderId + actionKey + eventDate` 当前态聚合, 重复 DONE 不让今日状态重复。
- Activity feed 保留事件审计, 当前态只看最新有效事件。

### P0-3 病历上传带来不必要的隐私风险

问题: 处方、化验单、影像等病历资料属于敏感个人健康资料, 上传和云端存储会显著提高隐私风险。  
优化:

- iOS 移除病历上传、附件上传和 OCR 病历夹入口。
- App 不再申请相机/相册权限。
- 服务端旧病历和附件接口统一返回 410 Gone。
- 同步墙和概览不再展示病历动态。

### P0-4 老人端复杂度可能过高

问题: 同一个 App 若直接暴露完整家庭管理功能, 老人会迷路。  
优化:

- 提供 Elder Mode, 首屏只显示今日动作、家人留言、下次复查。
- 家庭管理默认在 Caregiver tab。
- 支持大字号和 VoiceOver。

### P0-5 首版 Widget 范围过大影响交付

问题: 完整 Widget extension + App Group + App Intent + 后台同步会显著增加项目配置复杂度。  
优化:

- 首版代码预留 App Intent/Widget 设计, App 内先完成同等“一键动作”体验。
- 功能需求把 Widget 发布包列为延期, 但接口模型和 intent 设计不阻塞后续接入。

### P0-6 没有登录/家庭选择会导致数据归属不明确

问题: 如果 App 直接进入默认家庭, 用户无法确认自己是谁、属于哪个家庭, 后续多家庭/多老人会混乱。  
优化:

- App 首屏改为服务端登录, 提供“登录”和“注册”两个按钮。
- 账号密码使用 PBKDF2 哈希保存, 登录返回 Bearer token。
- Apple 登录服务端校验 Apple identity token issuer/audience。
- 登录后必须选择已加入家庭, 或创建/邀请码加入家庭。
- 登录时明确选择“我是老人/我是家人”, 老人身份再绑定当前老人档案。
- 所有服务端请求携带当前用户身份。

### P0-7 多老人家庭如果只显示单老人会误导照护状态

问题: 同一家庭可能同时照顾爸爸妈妈或双方父母, 单老人首页会让家庭以为所有老人都已覆盖。  
优化:

- 服务端模型和 overview 支持同一家庭多个 ElderProfile。
- iOS 同步墙默认展示“全部老人”, 并提供老人筛选。
- 老人模式可固定当前老人, 避免老人误点其他人的记录。

### P0-8 复查信息不完整会形成“空穴来风”的提醒

问题: 只有一句“复查”无法解释谁登记、去哪、带什么、是否有留言, 家庭成员无法执行。  
优化:

- 创建复查必须填写对象、时间、医院、科室、负责人和携带清单。
- 复查支持登记留言和完成留言。
- 同步墙和复查看板展示登记人/负责人头像与姓名。

## P1 问题

### P1-1 认证方式后续可增强

问题: 真实产品应使用 Sign in with Apple 或短信登录。  
优化: 首版已提供账号密码、Apple token 校验、未登录 401 和 Bearer token; 后续接入短信/OAuth2 时不改家庭资源模型。

### P1-2 推送通知未定义

问题: 子女希望“爸已吃药”实时收到通知。  
优化: 首版使用刷新/拉取; API 事件模型支持后续 APNs。

### P1-3 医疗记录上传已移除

问题: 原图、OCR 文本和结构化字段都会扩大敏感健康数据处理范围。  
优化: 不做病历上传和对象存储; 产品只保留低风险照护动作、复查和操作历史。

### P1-4 多老人/多家庭切换复杂

问题: 一个子女可能照顾双方父母。  
优化: 数据模型支持多家庭多老人; UI 首版显示一个当前家庭, 后续增加切换器。

### P1-5 日历和系统提醒未集成

问题: 复查日期需要进入系统日历。  
优化: 首版先在 App 内展示; 后续用 EventKit 由用户授权写入日历。

### P1-6 数据同步失败体验

问题: 网络差时老人可能看不懂失败状态。  
优化: 老人端统一显示“已记录, 正在同步”; 子女端显示最后同步时间。

## 优化后的 MVP 范围

必须做:

1. 家庭、成员、老人、照护动作、操作历史、复查任务和订阅权益的完整服务端闭环。
2. iOS 老人今日页 + 子女同步墙 + 操作历史/复查展示的核心 UI。
3. iOS 26 Liquid Glass 可用时启用, 非 iOS 26 fallback。
4. 服务端集成测试 + iOS 构建/模拟器 smoke test。

明确不做:

1. 医疗诊断、异常值判断、用药建议。
2. 生产级登录、APNs、对象存储、完整 Widget extension。
3. 病历上传、OCR 病历夹、医院系统集成。
